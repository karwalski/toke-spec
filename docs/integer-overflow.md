## 1. Default: Checked Arithmetic

All integer arithmetic operations on `i64` are **checked** by default. If an operation would overflow or underflow the 64-bit signed range (`-2^63` to `2^63 - 1`), the program traps with a runtime error.

### 1.1 Checked Operations

| Operation | Overflow Behavior |
|-----------|-------------------|
| `a + b` | Trap on signed overflow |
| `a - b` | Trap on signed underflow |
| `a * b` | Trap on signed overflow/underflow |
| `-a` (unary) | Trap if `a == -2^63` |
| `a / b` | Trap if `b == 0` (division by zero) or `a == -2^63 && b == -1` |
| `a % b` | Trap if `b == 0` |

### 1.2 Trap Behavior

On overflow:
1. Execution halts immediately (no undefined behavior, no silent wraparound)
2. A structured diagnostic is emitted: error code **RT002** ("integer overflow")
3. The diagnostic includes the operation, operand values, and source location
4. The process exits with a non-zero exit code

### 1.3 LLVM IR Emission

The compiler emits `llvm.sadd.with.overflow.i64` (and corresponding intrinsics for sub, mul) for checked operations:

```llvm
; Checked addition
%result = call {i64, i1} @llvm.sadd.with.overflow.i64(i64 %a, i64 %b)
%val = extractvalue {i64, i1} %result, 0
%overflow = extractvalue {i64, i1} %result, 1
br i1 %overflow, label %trap, label %continue

trap:
  call void @tk_overflow_trap(i64 %a, i64 %b, i32 <op_code>, i32 <line>)
  unreachable

continue:
  ; use %val
```

### 1.4 Optimization

At `-O2` and above, LLVM eliminates ~43% of overflow checks through range analysis (SPEC CPU2006 measured ~11.8% overhead at `-O1`, reducing to ~6.9% at `-O3`).

The compiler MAY add `nsw` annotations to operations where it can statically prove no overflow occurs (e.g., loop counters with known bounds).

## 2. Opt-Out: Wrapping Arithmetic

For performance-critical code where overflow behavior is intentional (hashing, bitwise operations, ring buffers), the `@wrapping` annotation opts out of overflow checks for a specific operation or block.

### 2.1 Per-Operation

```
let hash = a @wrapping+ b;
let rotated = x @wrapping* 0x9e3779b97f4a7c15;
```

### 2.2 Per-Block (future)

```
@wrapping {
  let h = seed;
  h = h * prime1;
  h = h + len;
  h = h * prime2;
};
```

### 2.3 LLVM IR for Wrapping

Wrapping operations emit plain `add i64` without overflow intrinsics:

```llvm
%result = add i64 %a, %b  ; no nsw, no overflow check
```

## 3. Rationale

### 3.1 Why Checked by Default?

toke programs are primarily generated by LLMs. LLMs cannot reason reliably about integer overflow conditions. Checked arithmetic:

- Catches bugs that would otherwise produce silently wrong results
- Eliminates an entire class of security vulnerabilities (CWE-190, CWE-191)
- Makes overflow failures explicit and debuggable via structured diagnostics
- Matches the safety philosophy of a language designed for code generation

### 3.2 Why Not Undefined Behavior?

Undefined behavior (C-style `add nsw`) would allow LLVM to optimize more aggressively, but:

- LLM-generated code is more likely to trigger overflow than human-written code
- UB turns overflow into unpredictable behavior (not just wrong results)
- The performance cost of checked arithmetic (~7-12%) is acceptable for correctness

### 3.3 Why @wrapping Instead of @unchecked?

The name `@wrapping` describes the semantics (two's complement wraparound), not the absence of checks. This makes intent clear at the call site.

## 5. Gate 1 Evidence

Checked overflow (D2=E) was specified in this document on 2026-04-04 and implemented in the reference compiler on the same date. Gate 1 evaluation (2026-04-03) was conducted entirely with **unchecked arithmetic** — all integer operations compiled to plain LLVM `add`/`sub`/`mul` instructions without overflow intrinsics.

Consequently:

- The Gate 1 Pass@1 result (63.7%) reflects unchecked arithmetic. No programs were rejected or trapped due to overflow.
- The Gate 1 token-efficiency result (12.5% same-tokenizer reduction) is unaffected, since checked overflow does not change source-level token counts.
- Future gates (Gate 2+) will measure the runtime overhead of checked arithmetic on the benchmark suite and report it alongside Pass@1. The expected overhead is 7-12% at `-O1`, decreasing at higher optimization levels (see Section 1.4).

## 6. Conformance Tests Required

The following test categories must be added:

1. **Boundary tests** — operations at `i64` min/max values
2. **Trap verification** — confirm program exits non-zero on overflow
3. **@wrapping correctness** — confirm wraparound produces expected results
4. **Optimization preservation** — confirm checked semantics survive `-O2`/`-O3`
5. **Diagnostic output** — confirm RT002 is emitted with correct fields
