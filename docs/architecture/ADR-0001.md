# ADR-0001: Initial Language and Compiler Architecture

**Date:** 2026-03-27
**Status:** accepted
**Deciders:** M. Karwalski

## Context

toke needs a foundational architecture decision before any implementation
begins. The language is designed for LLM code generation, not human authorship.
This ADR captures the initial architectural decisions that flow from that premise.

## Decision

1. **Two-phase character set:** Phase 1 (80 chars) is compatible with existing
   LLM tokenizers. Phase 2 (56 chars) uses a purpose-built BPE tokenizer for
   2.5–4x token density improvement. All initial work uses Phase 1.

2. **Reference compiler in C:** tkc is written in C99 with zero external
   runtime dependencies (LLVM used only for code generation, not at runtime).
   This minimises the build surface and maximises portability.

3. **LL(1) grammar:** The parser requires at most one token of lookahead.
   This eliminates ambiguity and makes the grammar mechanically verifiable.

4. **Structured JSON diagnostics:** All compiler errors are emitted as
   machine-readable JSON records with a stable schema. This is a load-bearing
   feature enabling automated generate-compile-repair loops.

5. **Arena allocation in compiler frontend:** All AST nodes are arena-allocated.
   No malloc/free in compiler frontend code outside of arena.c.

6. **Separate repositories per component:** Each major component has its own
   repository under github.com/karwalski to enable independent versioning
   and clear dependency tracking.

## Rationale

The generate-compile-repair loop is the core use case. Every architectural
decision is evaluated against: does this make the loop faster, more reliable,
or more automatable?

## Consequences

- The LL(1) constraint limits some language expressiveness but ensures the
  parser stays simple and the grammar stays unambiguous.
- The structured diagnostic requirement means every error path in the compiler
  must emit a schema-conforming record — no silent failures.
- The arena allocation model means the compiler frontend has predictable
  memory behaviour and no use-after-free bugs.

## Alternatives considered

- LALR(1) grammar: rejected — too much implementation complexity for too
  little expressive gain given the machine-generation use case.
- Single monorepo: rejected — makes independent versioning and licence
  management harder. The split-repo model allows toke-stdlib to be MIT
  while the compiler is Apache 2.0.
